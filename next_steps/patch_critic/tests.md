# Critic Orchestrator Tests

Run these tests locally to verify the Critic correctly evaluates diffs and controls the LangGraph state machine flow.

## 1. Critic Base Model Routing
- **Action**: Fire a test request to `evaluate_patch()` via `hf_tgi_chat("")` (empty adapter string).
- **Expected**: The request succeeds and the response confirms it was generated by the base `qwen2.5-coder:7b-instruct` model without applying any LoRA adapter layers.

## 2. False Positive Rejection
- **Action**: Provide the Critic with a diff that passes the Sandbox (`passed=True`), but logically does *not* address the User's Issue body (e.g., the issue asks to fix a timezone bug, but the diff adds a log print statement).
- **Expected**: The Critic outputs a JSON payload with `"verdict": "REJECT"` and explicitly mentions in the feedback that the core issue was ignored despite tests passing.

## 3. Valid Approval
- **Action**: Provide the Critic with a high-quality diff that passes tests and directly solves the mock issue.
- **Expected**: The Critic outputs `"verdict": "APPROVE"`.

## 4. LangGraph Edge Control
- **Action**: Run the full LangGraph state machine with a mock Critic that always returns `REJECT`.
- **Expected**: The graph loops exactly 3 times through `generate_patch` -> `run_sandbox` -> `critic_review` before gracefully hitting the `END` node and returning the last known state, proving the iteration cap works.

## 5. UI Reflection
- **Action**: Trigger `approve_plan` from the VS Code frontend.
- **Expected**: As the state machine loops, intermediate outputs or the final Critic verdict correctly marshal back through `app.py` and are rendered in the VS Code webview without breaking JSON parsing.
